Onboarding Diagnostic

First-week orientation framework · Service Booking Chatbot · 23 April 2026
Framing

This is the checklist I’d walk through in my first week to get oriented — not an assessment of what the team has built. Half of these items are probably already handled and I just don’t know it yet. The goal is to find where the real gaps are and where I can plug in, not to arrive with a verdict.

Four lenses — one critical item per lens

Measure

Do we know how we’re doing?

  1. Eval harness Critical Golden conversations covering happy paths and known edge cases; runs on every prompt or retrieval change; blocks regressions before rollout.
  2. Cost & latency budgets Per-turn token cost and p50/p95 latency tracked; alert thresholds wired up.
  3. Product KPIs Booking conversion, human-deflection rate, CSAT — sliced by cohort and intent.

Understand

What is actually happening in production?

  1. Conversation tracing Critical End-to-end turn traces (prompt, retrieved chunks, tool calls, completion) searchable by session; Langfuse / Application Insights / equivalent.
  2. Failure-mode clustering Weekly review of top failure categories — hallucinations, wrong tool choice, broken slot-filling, escalation misses.
  3. Intent coverage map Top-20 real user intents cross-tabbed against how well the bot handles each.

Core system quality

Is the brain working well?

  1. RAG retrieval quality Critical Precision & recall measured on golden queries; chunk-level eval; adversarial noise injection; separate scores for retrieval vs generation quality.
  2. Structured output discipline Pydantic schemas on every tool call; validation_error_rate tracked as a first-class metric.
  3. Prompt registry Versioned prompts in source control; every production response tagged with the prompt version that produced it.
  4. Context management Session state, vehicle context, token budget per turn — with explicit truncation and summarization rules.

Trust & safety

Can we deploy with confidence?

  1. Guardrails coverage Critical Jailbreak resistance, PII leakage prevention, out-of-scope deflection, brand-safety filters — each tested against a named adversarial suite.
  2. Propose → confirm → execute Enforced in code for every booking-adjacent action, not only at the UX layer. Covers cancellations, reschedules, and edge cases too.
  3. Fallback & escalation Confidence threshold → human handoff paths, end-to-end tested. Clear ownership of escalated conversations.
  4. Red-team & audit log Scripted adversarial runs before each release; complete audit trail of LLM decisions for post-incident review.

What I’d probe in the first conversations

Three things I’d expect are already in place. My first-week conversations would be about the specifics underneath them:

  • An eval harness exists. Question is what corpus it uses, how often it’s refreshed, and whether it reflects real user sessions or just the engineer’s imagination.
  • Guardrails are configured. Question is whether they’ve been adversarially tested recently against a current jailbreak set — or only once at setup.
  • A confirmation pattern exists in the UX. Question is whether the same invariant is enforced in code for every booking action — including reschedules, cancellations, and anything a future tool might add.
