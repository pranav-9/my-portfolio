Critical loop — booking actions: the LLM produces a typed proposed_booking object; orchestrator renders a confirmation message; the user says yes; only then does a deterministic handler call the booking API with an idempotency key. The LLM never writes to the booking system on its own.

Four design principles driving this

Grounded by default

If the bot states a service code, price, wait time, or policy, it must come from a retrieved source. No free-form reasoning on facts the business cares about.

Propose → confirm → execute

No LLM call is ever allowed to directly commit a booking. Proposals are typed objects; execution is deterministic code with idempotency keys and retries.

Dual state

Session state (short-term, per-conversation) and vehicle state (long-term, per-VIN) are separate stores. Telematics signals inject as structured tool results, not free text.

Eval harness from day one

A regression suite of scripted conversations (happy path, edge cases, adversarial) runs on every prompt change. No blind rollouts.

Assumptions I’d want to validate early

  • Booking API surface: full CRUD or read-only? Does it support tentative holds and idempotency keys?
  • Telematics: real-time stream, batch, or polled? Which diagnostic signals surface to the chatbot layer today?
  • Corpus: is there an existing service-content index, or does the RAG corpus need assembly from scratch?
  • Success metric: booking conversion, agent deflection, CSAT, or a composite?
